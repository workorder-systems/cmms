# Errors

When a request fails, the SDK throws an `SdkError` (or rethrows it if the error was already an `SdkError`). It wraps PostgREST/Supabase errors and preserves `code`, `details`, and `hint` for debugging. {{ className: 'lead' }}

## SdkError

All resource methods and `setTenant` / `clearTenant` normalize failures into `SdkError`:

<Properties>
  <Property name="message">
    Error message from PostgREST or RPC.
  </Property>
  <Property name="code">
    Optional. PostgREST/Supabase error code (e.g. `PGRST301`, auth codes).
  </Property>
  <Property name="details">
    Optional. Additional detail from the backend.
  </Property>
  <Property name="hint">
    Optional. Hint from the backend on how to fix the issue.
  </Property>
</Properties>

<CodeGroup title="Catching SdkError">

```ts
import { createDbClient, SdkError } from '@workorder-systems/db'

const client = createDbClient(url, anonKey)

try {
  await client.workOrders.list()
} catch (err) {
  if (err instanceof SdkError) {
    console.error(err.message, err.code, err.details, err.hint)
  }
  throw err
}
```

</CodeGroup>

You can also use `normalizeError(error)` to turn an unknown thrown value into an `SdkError` without throwing.

## Common codes

Errors come from Supabase/PostgREST and your RPCs. Typical sources:

- **PGRST*** ÔÇö PostgREST (e.g. invalid filter, missing column). Check [PostgREST errors](https://postgrest.org/en/stable/errors.html) for code meanings.
- **Auth** ÔÇö Supabase Auth (e.g. invalid credentials, session expired). Check [Supabase Auth docs](https://supabase.com/docs/guides/auth) for handling.
- **RPC** ÔÇö Your `rpc_*` functions can return custom messages and codes; `details` and `hint` may contain server-set text.

## Status codes

The SDK does not expose HTTP status codes directly; it throws on error. Under the hood, Supabase uses standard HTTP: 2xx for success, 4xx for client errors (e.g. 401 unauthorized, 403 forbidden, 404 not found), 5xx for server errors. Use `SdkError.message` and `code` / `details` / `hint` to decide how to handle failures in your app.

**Permission denied (42501):** When an RPC requires a permission or scope the user does not have, the backend returns an error (often with a message like ÔÇťPermission denied ÔÇŽ tenant.admin requiredÔÇŁ or ÔÇťUnauthorized ÔÇŽ workorder.editÔÇŁ). See [Authorization (RBAC & ABAC)](/authorization) for how roles, permissions, and scopes work.
