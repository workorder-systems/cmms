# Authentication

Sign in with Supabase Auth. Every request runs as the authenticated user—no separate API key header. The SDK uses your Supabase session. {{ className: 'lead' }}

## How it works

- **Auth:** Supabase Auth (JWT). You use the project URL and **anon** key to create the client. The anon key is safe in the browser; RLS and RPC permission checks enforce what each user can do.
- **Session:** Sign in with `client.supabase.auth.signInWithPassword()` (or OAuth, magic link, and so on). The SDK’s tenants, work orders, and other resources use the same client, so they automatically use the current session.

## Sign in

Get your project URL and anon key from the [Supabase dashboard](https://supabase.com/dashboard) (Project Settings → API). Create the client and sign in:

<CodeGroup title="Sign in with email and password">

```ts
import { createDbClient } from '@workorder-systems/db'

const client = createDbClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_ANON_KEY!
)

const { data, error } = await client.supabase.auth.signInWithPassword({
  email: 'user@example.com',
  password: 'your-password',
})
```

</CodeGroup>

After sign-in, the JWT is stored in the session (or in whatever storage you configured). All later calls to `client.tenants.list()`, `client.workOrders.list()`, and so on use this session.

## Using the SDK

You don't pass a token on each request. The SDK wraps the Supabase client; once the user is signed in via `client.supabase.auth`, every resource method runs as that user. For tenant-scoped data you also need to call `client.setTenant(tenantId)` and refresh the session so the JWT includes tenant context. See [Tenant context](/tenant-context).
