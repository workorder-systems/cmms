# Work orders

Lists and gets work orders for the current tenant; create, transition status, complete, log time, add attachments. Set tenant context before calling these methods. {{ className: 'lead' }}

## List work orders

List all work orders for the current tenant (from `v_work_orders`).

<CodeGroup title="client.workOrders.list()">

```ts
await client.setTenant(tenantId)
// Refresh session so JWT carries tenant_id, then:
const workOrders = await client.workOrders.list()
// Returns WorkOrderRow[]
```

</CodeGroup>

## Get by ID

Fetch a single work order by ID.

<CodeGroup title="client.workOrders.getById(id)">

```ts
const workOrder = await client.workOrders.getById('uuid-of-work-order')
// Returns WorkOrderRow | null
```

</CodeGroup>

## Create work order

Create a new work order. Returns the new work order UUID.

<CodeGroup title="client.workOrders.create(params)">

```ts
const workOrderId = await client.workOrders.create({
  tenantId: 'uuid-of-tenant',
  title: 'Repair HVAC unit',
  description: 'Inspect and repair unit 3B',
  priority: 'medium',
  maintenanceType: 'corrective',
  assignedTo: 'uuid-of-user',
  locationId: 'uuid-of-location',
  assetId: 'uuid-of-asset',
  dueDate: '2025-03-01',
  pmScheduleId: null,
})
// Returns string (UUID)
```

## Attachments overview

Attachments use **Supabase Storage** and a **database trigger**: there is no RPC to “add” an attachment. You upload a file to the `attachments` bucket with a path that includes `tenant_id` and `work_order_id`. A trigger then creates the file record in `app.files` and the link in `app.work_order_attachments`. You list attachments via the view `v_work_order_attachments` and get short-lived **signed URLs** from `bucket_id` and `storage_path` (no permanent URLs are stored).

| Action | How |
|--------|-----|
| Create attachment | Upload to `attachments` bucket (path: `tenant_id/work_order_id/uuid_filename`); trigger creates records. |
| List attachments | Query `v_work_order_attachments` (with tenant context set). |
| Get file URL | `client.supabase.storage.from(bucket_id).createSignedUrl(storage_path, expiresIn)`. |
| Update label/kind | RPC `rpc_update_work_order_attachment_metadata(attachment_id, label?, kind?)`. |

Use `client.supabase` for Storage and for querying the view; set tenant context with `client.setTenant(tenantId)` before listing or uploading.

## Upload attachment

Upload the file to the `attachments` bucket. The path must be `{tenantId}/{workOrderId}/{uuid}_{filename}` so the trigger can associate the file with the work order. Optional metadata (`work_order_id`, `tenant_id`, `label`, `kind`) may be passed if your Supabase version supports it; otherwise set label/kind later via `rpc_update_work_order_attachment_metadata`.

<CodeGroup title="Upload to attachments bucket">

```ts
await client.setTenant(tenantId)

const path = `${tenantId}/${workOrderId}/${crypto.randomUUID()}_${file.name}`
const { data, error } = await client.supabase.storage
  .from('attachments')
  .upload(path, file, {
    contentType: file.type ?? 'application/octet-stream',
    upsert: false,
    metadata: {
      work_order_id: workOrderId,
      tenant_id: tenantId,
      label: 'Before photo',  // optional
      kind: 'photo',          // optional
    },
  })
if (error) throw error
// Trigger creates app.files + work_order_attachments; no attachment UUID returned from upload.
// To get the new attachment id, list v_work_order_attachments for this work_order_id and take the latest.
```

</CodeGroup>

The uploader must be a tenant member and either **assigned to the work order** or have **workorder.edit** permission; otherwise Storage RLS rejects the upload.

## List attachments

Query the view `v_work_order_attachments` with tenant context set. The view exposes `id`, `work_order_id`, `file_id`, `bucket_id`, `storage_path`, `filename`, `label`, `kind`, `created_by_name`, `created_at`.

<CodeGroup title="List attachments for a work order">

```ts
await client.setTenant(tenantId)

const { data: attachments, error } = await client.supabase
  .from('v_work_order_attachments')
  .select('id, work_order_id, file_id, bucket_id, storage_path, filename, label, kind, created_by_name, created_at')
  .eq('work_order_id', workOrderId)
  .order('created_at', { ascending: false })

if (error) throw error
// Use bucket_id and storage_path to create signed URLs (see next section).
```

</CodeGroup>

## Get signed URL

Attachments are stored in a **private** bucket. Do not store permanent URLs. For each attachment row, call `createSignedUrl` with `bucket_id` and `storage_path` from the view to get a short-lived URL (e.g. 1 hour). RLS on `storage.objects` is evaluated when creating the signed URL; only users who can read the linked work order receive a URL.

<CodeGroup title="Create signed URL for an attachment row">

```ts
const { data, error } = await client.supabase.storage
  .from(row.bucket_id)   // from v_work_order_attachments
  .createSignedUrl(row.storage_path, 3600)  // 3600 seconds = 1 hour

if (error) throw error
// data.signedUrl — use for display or download
```

</CodeGroup>

## Update attachment metadata

To set or change the label and/or kind after upload (e.g. when Storage metadata was not used), call `rpc_update_work_order_attachment_metadata`. The caller must be a tenant member and either the attachment creator or an admin/manager.

<CodeGroup title="rpc_update_work_order_attachment_metadata">

```ts
await client.supabase.rpc('rpc_update_work_order_attachment_metadata', {
  p_attachment_id: attachmentId,  // from v_work_order_attachments
  p_label: 'Before photo',
  p_kind: 'photo',
})
```

</CodeGroup>

<Properties>
  <Property name="p_attachment_id">
    Required. Attachment UUID (view row <code>id</code>).
  </Property>
  <Property name="p_label">
    Optional. Label. Max 255 characters.
  </Property>
  <Property name="p_kind">
    Optional. Kind key (e.g. <code>photo</code>, <code>document</code>); format a-z0-9_.
  </Property>
</Properties>
